# XTracker Elon 歷史資料初始化交接

## 背景

這個 repo 是 `C:\work\data-treasure`，目前已經有 XTracker crawler 模組，可以呼叫 `https://xtracker.polymarket.com/api/users/{handle}/posts` 抓取人物貼文、保存 raw snapshot，並 upsert normalized post。

目標是評估並實作一組「資料初始化 / backfill」流程：先往前探測 XTracker 最早有資料的時間，再把歷史資料分段慢慢補進 DB。

## 已確認的市場時間規則

使用者一開始要看 Polymarket market：

> Elon Musk (@elonmusk) 在 May 15 12:00 PM ET 到 May 22, 2026 12:00 PM ET 的 X 發文數。

注意：May 2026 的 ET 是 EDT，也就是 UTC-4。

所以正確 UTC 時窗是：

- market start: `2026-05-15T16:00:00Z`
- market end: `2026-05-22T16:00:00Z`

目前程式/先前 sync 用過 `2026-05-15T04:00:00Z` 到 `2026-05-22T04:00:00Z` 並帶 `timezone=EST`，這不符合該 market 的 `12:00 PM ET`。

建議後續全部用 UTC 半開區間 `[start, end)`，不要依賴 `timezone=EST`。

## 目前 DB 狀態

本機 Docker 有 Postgres：

- container: `data_treasure_postgres`
- DB: `data_treasure`
- user: `data_treasure`

已查到 DB 中 Elon 記錄：

- `platform = X`
- `handle = elonmusk`
- `last_seen_at = 2026-05-22 08:01:52.380486+00`
- normalized total posts: `285`
- DB 中最早 `posted_at = 2026-05-15 04:22:37+00`
- DB 中最晚 `posted_at = 2026-05-22 07:50:15+00`

用正確 market 時窗查 DB：

```sql
select count(*) as market_window_count,
       min(cp.posted_at) as first_posted_at,
       max(cp.posted_at) as last_posted_at
from xtracker_crawled_person p
join xtracker_crawled_post cp on cp.person_id = p.id
where p.platform = 'X'
  and p.handle = 'elonmusk'
  and cp.posted_at >= timestamptz '2026-05-15 16:00:00+00'
  and cp.posted_at <  timestamptz '2026-05-22 16:00:00+00';
```

結果：

- `market_window_count = 266`
- first = `2026-05-15 16:09:15+00`
- last = `2026-05-22 07:50:15+00`

但因為目前 DB 只爬到 `2026-05-22 08:01:52Z` 左右，market end 到 `2026-05-22T16:00:00Z` 的後段還沒完整。

## XTracker API 探測結果

PowerShell `Invoke-RestMethod` 會遇到憑證或 403 問題，但 `curl.exe -k` 可成功抓取。

可用命令範例：

```powershell
curl.exe -k "https://xtracker.polymarket.com/api/users/elonmusk/posts?platform=X&startDate=2026-05-15T16%3A00%3A00Z&endDate=2026-05-22T16%3A00%3A00Z"
```

API response 形狀：

```json
{
  "success": true,
  "data": [
    {
      "id": "cmpgmjwxi0006ic04lt27s8ct",
      "userId": "c4e2a911-36ec-4453-8a39-1edb5e6b2969",
      "platformId": "2057730745821483416",
      "content": "RT @tetsuoai: https://t.co/RFO8ZtukZc",
      "createdAt": "2026-05-22T07:50:15.000Z",
      "importedAt": "2026-05-22T07:55:08.023Z",
      "metrics": null
    }
  ]
}
```

重要：`id` 是 XTracker internal id，`platformId` 才是真正 X status id。

目前程式 `XTrackerPostParseService` 會先抓 `id` 當 `sourcePostId`，導致 normalized `source_post_id` 是 `cmp...`，自動產生的 `post_url` 會變成假的：

```text
https://x.com/elonmusk/status/cmp...
```

後續初始化前建議先修這件事。

## 目前程式相關檔案

- `src/main/java/io/github/yienruuuuu/xtracker/service/XTrackerFetchService.java`
  - 呼叫 XTracker API。
  - endpoint: `https://xtracker.polymarket.com/api/users/{handle}/posts`
  - query params: `platform`, `startDate`, `endDate`, `timezone`

- `src/main/java/io/github/yienruuuuu/xtracker/service/XTrackerPostParseService.java`
  - 解析 response。
  - 目前 `sourcePostId` 欄位順序是 `id`, `postId`, `post_id`, `tweetId`, `tweet_id`, `sourcePostId`。
  - 應考慮改成優先 `platformId`，並保留 tracker internal id。

- `src/main/java/io/github/yienruuuuu/xtracker/service/XTrackerPersonSyncService.java`
  - orchestrates one sync。
  - 會保存 raw snapshot，解析 posts，upsert normalized posts。

- `src/main/java/io/github/yienruuuuu/xtracker/dao/XTrackerCrawledPostDao.java`
  - count 邏輯是 `postedAt >= startAt` 且 `postedAt < endAt`，這個半開區間方向正確。

- `src/main/resources/common-script/V1.0.4__xtracker_crawler.sql`
  - 目前 schema 沒有 `platform_post_id`、`tracker_post_id`、`imported_at`、`post_type` 等欄位。
  - 如果要改 schema，依 AGENTS.md 規則新增 Flyway migration，不要改已套用 migration。

## 建議下一步

### 1. 先修 normalized post identity

新增 migration，讓 normalized post 可以同時保存：

- `tracker_post_id`: XTracker internal `id`
- `platform_post_id`: X 真正 status id，也就是 response `platformId`
- `imported_at`: XTracker 捕捉時間

可考慮保留既有 `source_post_id` 作相容，但後續真正 dedupe key 建議改為 `(person_id, platform_post_id)`，或至少在 parser 中讓 `source_post_id = platformId`。

如果要避免大改 DB，可以短期先把 parser 改成：

- `sourcePostId` 優先吃 `platformId`
- `postUrl = https://x.com/{handle}/status/{platformId}`
- raw snapshot 照樣保存完整 JSON

但這會和目前 DB 既有 `cmp...` rows 產生重複資料風險，需要 migration 或清理策略。

### 2. 實作 backfill/discovery service

建議新增 service，例如：

- `XTrackerBackfillDiscoveryService`
- `XTrackerBackfillTaskService`

流程：

1. 從指定 `endAt` 往前按月查。
2. 找到第一個有資料月份。
3. 對該月按天查。
4. 確定 earliest `createdAt`。
5. 從 earliest 所在日開始建立一批 backfill windows。

窗口建議：

- 初期按日，降低 response 過大與重試成本。
- 穩定後可按月。

### 3. 用 scheduler task 慢慢補資料

不要一次塞很大的 while loop。

建議每個 window 對應一個 persistent scheduled task，payload 類似：

```json
{
  "platform": "X",
  "handle": "elonmusk",
  "startAt": "2025-10-31T00:00:00Z",
  "endAt": "2025-11-01T00:00:00Z",
  "forceRawSnapshot": false
}
```

handler 呼叫既有 `XTrackerPersonSyncService.syncOnce(platform, handle, options)`。

### 4. 市場查詢 API 使用 UTC

市場 `May 15 12:00 PM ET` 到 `May 22 12:00 PM ET` 應查：

```text
startAt=2026-05-15T16:00:00Z
endAt=2026-05-22T16:00:00Z
```

不要再用 `timezone=EST` 表示市場時間。

## 可重跑的探測 PowerShell

按月掃描：

```powershell
$base='https://xtracker.polymarket.com/api/users/elonmusk/posts'
$from=[datetimeoffset]'2025-01-01T00:00:00Z'
$to=[datetimeoffset]'2026-06-01T00:00:00Z'
$rows=@()
for($s=$from; $s -lt $to; $s=$s.AddMonths(1)){
  $e=$s.AddMonths(1)
  $url=$base + '?platform=X&startDate=' + [uri]::EscapeDataString($s.ToString('yyyy-MM-ddTHH:mm:ssZ')) + '&endDate=' + [uri]::EscapeDataString($e.ToString('yyyy-MM-ddTHH:mm:ssZ'))
  $json=& curl.exe -k -sS $url
  $r=$json | ConvertFrom-Json
  $items=@($r.data)
  $dates=@($items | ForEach-Object { $_.createdAt } | Sort-Object)
  $rows += [pscustomobject]@{
    Month=$s.ToString('yyyy-MM')
    Count=$items.Count
    Min=($dates | Select-Object -First 1)
    Max=($dates | Select-Object -Last 1)
  }
  Start-Sleep -Milliseconds 150
}
$rows | Format-Table -AutoSize
```

按天掃描 2025-10：

```powershell
$base='https://xtracker.polymarket.com/api/users/elonmusk/posts'
$from=[datetimeoffset]'2025-10-01T00:00:00Z'
$to=[datetimeoffset]'2025-11-01T00:00:00Z'
$rows=@()
for($s=$from; $s -lt $to; $s=$s.AddDays(1)){
  $e=$s.AddDays(1)
  $url=$base + '?platform=X&startDate=' + [uri]::EscapeDataString($s.ToString('yyyy-MM-ddTHH:mm:ssZ')) + '&endDate=' + [uri]::EscapeDataString($e.ToString('yyyy-MM-ddTHH:mm:ssZ'))
  $json=& curl.exe -k -sS $url
  $r=$json | ConvertFrom-Json
  $items=@($r.data)
  $dates=@($items | ForEach-Object { $_.createdAt } | Sort-Object)
  if($items.Count -gt 0){
    $rows += [pscustomobject]@{
      Day=$s.ToString('yyyy-MM-dd')
      Count=$items.Count
      Min=($dates | Select-Object -First 1)
      Max=($dates | Select-Object -Last 1)
    }
  }
  Start-Sleep -Milliseconds 100
}
$rows | Format-Table -AutoSize
```
